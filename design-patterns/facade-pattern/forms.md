# Facade 模式 - 诊断与规划表

## 第1步: 需求诊断 - 你真的需要Facade吗？

### 🔍 快速检查清单

```
□ 系统拥有3个+复杂子系统需要客户端协调
□ 子系统之间存在复杂的相互依赖关系
□ 客户端需要理解多个类的协作流程
□ 子系统接口不够友好，需要再次包装
□ 多个不同的客户端以相同方式使用子系统
□ 系统内部改动频繁，希望与外界隔离
```

**诊断标准**:
- ✅ 勾选 5 项以上 → **强烈推荐使用Facade**
- ⚠️ 勾选 3-4 项 → **可以使用Facade**
- ❌ 勾选 2 项以下 → **可能不需要，直接使用子系统**

---

## 第2步: 子系统复杂度评估

### 📊 子系统分析矩阵

| 维度 | 评分标准 | 我的情况 | 权重 |
|------|---------|---------|------|
| **类数量** | 1-3个=低, 4-6个=中, 7+个=高 | _____ | 20% |
| **相互依赖** | 无依赖=低, 链式=中, 环形=高 | _____ | 25% |
| **接口复杂度** | 参数<3=低, 3-5=中, 5+=高 | _____ | 20% |
| **使用频率** | 一处=低, 3+ 处=高 | _____ | 15% |
| **变化频率** | 不变=低, 定期改动=高 | _____ | 20% |

**评分**: 
- 总分 < 25 分: 简单，无需Facade
- 总分 25-50 分: 中等，单层Facade足够
- 总分 > 50 分: 复杂，可能需多层Facade

## 第3步: Facade类型选择矩阵

### 🎯 Facade设计决策树

```
开始 → 客户端数量?
   ├─ 单个客户端 + 简单流程 → 不需要Facade
   ├─ 单个客户端 + 复杂流程(3-5步)
   │   └─ → 单一Facade ✓
   └─ 多个客户端 或 5+步骤?
       ├─ 子系统分组清晰?
       │   └─ YES → 多层Facade ✓
       ├─ 各客户端需求差异大?
       │   └─ YES → 多Facade ✓
       └─ 需要渐进式重构?
           └─ YES → 分层Facade ✓
```

### 🔄 Facade类型对比表

| Facade类型 | 复杂度 | 灵活性 | 维护性 | 最适用 |
|-----------|--------|--------|--------|--------|
| **单一Facade** | ⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ | 子系统<5个,流程简单 |
| **多层Facade** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | 子系统分组,客户端多 |
| **多Facade** | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | 客户端需求差异大 |
| **Facade+Factory** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 高度定制化系统 |

## 第4步: 实现规划 - 6步快速上手

### 📋 实现步骤模板

**Step 1: 识别子系统边界** (15分钟)
```
□ 列出所有涉及的类: ________________________
□ 标记类之间的依赖关系
□ 确定最少必需的"暴露"方法
□ 例: {Auth, DB, Email} → {validateUser, saveUser, notifyUser}
```

**Step 2: 设计Facade职责** (20分钟)
```
□ 定义Facade需要支持的主要用例
□ 为每个用例设计一个public方法
□ 方法名反映业务含义，不暴露技术细节
□ 例: register() 而非 callAuthThenDBThenEmail()
```

**Step 3: 隐藏实现细节** (15分钟)
```
□ Facade内的对象应该private final
□ 子系统类不应被外界直接access
□ 使用包级别访问控制
□ 验证: 客户端只通过Facade使用子系统
```

**Step 4: 处理异常与边界** (20分钟)
```
□ 定义Facade方法的异常处理策略
□ 子系统异常转换为Facade异常
□ 考虑统一异常简化客户端处理
□ 例: FacadeException代替{AuthException, DBException}
```

**Step 5: 版本变化处理** (15分钟)
```
□ 子系统升级时如何不影响Facade客户端?
□ 考虑使用Adapter包装新旧版本
□ 在Facade定义稳定的接口契约
□ 例: 升级时添加新method而不修改旧method
```

**Step 6: 文档与示例** (10分钟)
```
□ 为Facade公有方法提供JavaDoc
□ 解释方法做了什么（不是怎么做的）
□ 提供完整的使用示例代码
□ 列出支持的异常类型
```

## 第5步: 测试计划

### ✅ 单元测试检查清单

```
□ Mock所有子系统依赖（隔离测试Facade）
□ 测试Facade方法的主流程是否正常
□ 测试异常场景: 单个子系统失败
□ 测试异常场景: 多个子系统失败
□ 测试原子性: 部分步骤失败后是否状态一致

□ 验证Facade隐藏了复杂性
□ 性能测试: 额外层是否产生明显开销
```

### 🧪 集成测试检查清单

```
□ 使用真实子系统实例测试端到端流程
□ 验证子系统间的数据流通是否正确
□ 测试子系统的真实异常处理
□ 验证Facade的功能与文档一致
```

## 第6步: 代码审查清单 - 12项检查

### 🔍 Facade质量检查

- [ ] **单一职责**: Facade只处理协调，不包含业务逻辑
- [ ] **隐藏性**: 所有子系统类为private，不暴露给客户端
- [ ] **接口简洁**: public方法数<10个（通常3-7个）
- [ ] **方法命名**: 名称反映业务概念，不暴露实现细节
- [ ] **异常处理**: 统一异常策略，客户端不需处理多种异常
- [ ] **文档完整**: 每个public方法都有清晰的JavaDoc
- [ ] **涉及状态**: 无状态或状态简单且清晰管理
- [ ] **线程安全**: 如果在多线程环境使用
- [ ] **性能**: 额外开销<5%
- [ ] **易扩展**: 添加新子系统只需修改Facade而不改客户端
- [ ] **避免重复**: 没有在子系统间重复协调逻辑
- [ ] **恰当粒度**: 没有为了Facade而Facade

## 第7步: 常见陷阱与预防

### ⚠️ 陷阱1: Facade变成上帝类

**问题**: Facade包装所有方法，变得巨大（>50个method）

**预防**:
- ✅ 按功能域拆分为多个Facade
- ✅ 例: UserFacade, PaymentFacade, NotificationFacade

### ⚠️ 陷阱2: 暴露了太多子系统细节

**问题**: 客户端仍然需要理解子系统

**预防**:
- ✅ 返回简单的基本类型或DTO/VO
- ✅ 例: pay(String userId, double amount) 而非 pay(User, Database, PaymentService)

### ⚠️ 陷阱3: Facade = 简单Wrapper

**问题**: Facade只是简单转发，没有真正简化

**预防**:
- ✅ Facade要处理协调、异常、边界条件
- ✅ 添加有业务价值的协律逻辑

### ⚠️ 陷阱4: Facade引入过度复杂

**问题**: 为了灵活性引入过多配置或DI依赖

**预防**:
- ✅ Facade应该自己创建和管理子系统
- ✅ 保持初始化简单

## 快速参考与启动模板

### 🚀 30秒Facade模板

```java
// 子系统
class Service1 { public void work1() {} }
class Service2 { public void work2() {} }

// Facade
class SimpleFacade {
    private Service1 s1 = new Service1();
    private Service2 s2 = new Service2();
    
    public void complexOp() {
        s1.work1();
        s2.work2();
    }
}

// 使用
new SimpleFacade().complexOp();
```
