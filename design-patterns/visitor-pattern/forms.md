# Visitor 模式 - 实施表单与诊断

## 何时使用 Visitor 模式诊断表

| 问题场景 | 指标 | Visitor 适配 | 替代方案 |
|---------|------|-------------|---------|
| 多类型对象集合 | 3+ 元素类型 | ✅ 是 | 多态 |
| 频繁添加操作 | 操作数 > 元素类型数 | ✅ 是 | 反射/Type |
| 元素结构稳定 | 结构变更频率 | ✅ 低频合适 | 策略模式 |
| 跨元素关联操作 | 单个元素类型无法完成 | ✅ 是 | 策略 |
| 性能关键 | 访问频率 > 100k/s | ⚠️ 谨慎 | 直接调用 |

## Visitor 类型快速参考

### Classical Visitor（推荐基础用法）

```java
interface Visitor {
    void visitA(TypeA a);
    void visitB(TypeB b);
}

class MyVisitor implements Visitor {
    public void visitA(TypeA a) { /* ... */ }
    public void visitB(TypeB b) { /* ... */ }
}
```

**何时使用**: 元素类型数量固定, 操作明确

### Parameterized Visitor（灵活操作）

```java
interface Visitor<T> {
    T visitA(TypeA a);
    T visitB(TypeB b);
}

class SumVisitor implements Visitor<Integer> {
    public Integer visitA(TypeA a) { return a.value; }
}
```

**何时使用**: 需要返回不同类型结果

### Functional Visitor（现代 Java）

```java
Map<Class<?>, Function<Object, Result>> visitors = Map.of(
    TypeA.class, obj -> processA((TypeA) obj),
    TypeB.class, obj -> processB((TypeB) obj)
);
```

**何时使用**: 简单操作, 动态扩展

## 缺陷预防检查表

### 访问方法爆炸

**症状**: 50+ 个 visit 方法

**诊断**:
- [ ] 元素类型数量 > 20
- [ ] 每个类型平均 3+ 操作

**解决方案**:
- [ ] 分组相关元素
- [ ] 使用 Parameterized 变体
- [ ] 切换到函数式方法

### 结构变更成本

**症状**: 添加新元素类型需修改所有 Visitor 实现

**诊断**:
- [ ] 合力变更超过 5 个文件
- [ ] 冒烟测试失败

**解决方案**:
- [ ] 添加默认 visit 方法
- [ ] 实现 Null Visitor 基类
- [ ] 使用接口默认方法

### 性能不足

**症状**: 访问 > 1M 对象时变慢

**诊断**:
- [ ] 虚调用开销
- [ ] 对象分配过多
- [ ] 内存缓存失效

**解决方案**:
- [ ] 结果缓存（LRU）
- [ ] 对象池回收
- [ ] 并行流处理

## 元素变更影响分析

添加新元素类型时评估影响范围:

| 元素 | 现有Visitor数 | 需改动Visitor个 | 需添加方法 | 预计时间 | 风险等级 |
|------|-------------|-------------|----------|---------|---------|
| 新建TypeD | 5个 | 5个 | visitTypeD() | 4小时 | 中 |
| 现有Element扩展 | 5个 | 3个 | findNewField() | 2小时 | 低 |
| 优化操作 | 5个 | 1个 | optimizeVisit() | 1小时 | 低 |

**评估规则**:
- 如果受影响Visitor > 10个 → 考虑策略模式替代
- 如果每个Visitor改动 > 3处 → 重构访问接口
- 风险等级:
  - 低: 变更 ≤ 1个文件
  - 中: 变更 2-5个文件  
  - 高: 变更 > 5个文件

## 性能规划与优化表

预估Visitor性能指标:

| 场景 | 元素数量 | 访问类型 | 预期延迟 | 吞吐量 | 优化方向 |
|------|---------|---------|---------|--------|---------|
| 小集合验证 | < 100 | 单遍 | < 1ms | 100k/s | 无需优化 |
| 中等报告生成 | 100-10k | 多遍 | 10-100ms | 10k/s | 缓存中间结果 |
| 大规模统计 | 10k-100k | 聚合 | 100-500ms | 1k/s | 并行 + 流式 |
| 超大数据转换 | > 100k | 转换 | > 1s | < 1k/s | 分片处理 + 缓存 |

**优化建议**:
1. **缓存策略**: 
   - 访问频率高 → LRU缓存
   - 结果固定 → 预计算缓存
   
2. **并发优化**:
   - 无状态Visitor → 线程池执行
   - 有状态Visitor → ThreadLocal隔离

3. **内存优化**:
   - 大量结果聚合 → 流式输出
   - 临时对象 → 对象池

## 快速决策表

**问题**: 应该用Visitor还是其他模式?

| 情况 | 使用Visitor | 替代方案 | 理由 |
|------|----------|---------|------|
| 元素3个, 操作5+ 个 | ✅是 | - | Visitor优势明显 |
| 元素5个, 操作1个 | ❌否 | 多态 | 过度设计 |
| 元素稳定, 操作变化 | ✅是 | - | 完美匹配 |
| 元素频繁变更 | ❌否 | 策略 | 维护成本高 |
| 需要访问私有字段 | ⚠️需检查 | 友元类 | 可行但权限复杂 |
| 跨语言操作 | ⚠️需检查 | 序列化 | 可行但协议复杂 |
