# Adapter Pattern - 诊断与规划指南

## 第1步: 不兼容性诊断矩阵

评估接口不兼容程度：

```
场景              是否使用  参数映射  返回值差  异常处理  文档齐全
                第三方库   复杂度    异  
──────────────────────────────────────────────────────────
新库vs旧API        ✓        高       有     有      否    → 用适配器
数据格式转换       ✓        中       有     无      是    → 用转换器
接口风格差异       ✓        低       无     有      是    → 用包装器
类型系统差异       ✓        高       有     有      是    → 用适配器
版本不兼容         ✓        中       有     有      否    → 多版本支持
```

### 快速诊断 (勾选3项表示推荐用适配器)

- [ ] 需要集成第三方库但接口不兼容
- [ ] 旧系统与新系统接口差异大
- [ ] 不能修改原有类的源代码
- [ ] 接口转换逻辑复杂需要集中管理
- [ ] 支持多个兼容对象同时存在

## 第2步: 适配器类型选择决策图

```
需要适配不兼容接口?
├─ 否 → 直接使用/修改源类
└─ 是 → 使用单继承还是多继承?
   ├─ 单继承 → 使用类适配器
   └─ 多接口 → 使用对象适配器 ⭐
        └─ 需支持多版本?
           ├─ 是 → 策略+适配器组合
           └─ 否 → 简单对象适配器
```

### 实现方法对比

| 方法 | 适用场景 | 灵活性 | 复杂度 |
|------|---------|--------|--------|
| 类适配 | 单继承可行 | ⭐⭐ | ⭐⭐ |
| 对象适配 | 一般场景 | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 双向适配 | 两向转换 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 装饰改进 | 需增强功能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

## 常见陷阱预防

### ⚠️ 陷阱1: 过度适配导致复杂度爆炸
```
❌ 错误: 为每个版本创建新适配器, 类数增加
✅ 修复: 使用策略模式支持多版本
```

### ⚠️ 陷阱2: 适配器泄露源类细节
```
❌ 错误: 抛出源类异常给客户端
✅ 修复: 统一转换为目标异常类型
```

### ⚠️ 陷阱3: 版本升级适配器失效
```
❌ 错误: 适配器硬编码版本号
✅ 修复: 动态版本检测和多版本支持
```

### ⚠️ 陷阱4: 性能问题
```
❌ 错误: 每次调用都创建新适配器
✅ 修复: 单例或缓存适配器实例
```

## 快速参考

### MVP 最小实现
```java
public class Simple Adapter implements TargetInterface {
    private SourceClass source;
    
    public SimpleAdapter(SourceClass source) {
        this.source = source;
    }
    
    @Override
    public void targetMethod() {
        source.sourceMethod();
    }
}
```

### 多版本支持
```java
TargetInterface adapter = version.equals("1.0") ? 
    new V1Adapter(source) : 
    new V2Adapter(source);
```

| 项目 | 耗时 |
|------|------|

| 单次调用 | <0.02ms |
| 100万调用 | <20s |
