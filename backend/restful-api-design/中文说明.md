# RESTful API设计与实现

## 目的
教学如何设计和实现高质量的RESTful API，遵循REST原则和最佳实践。

## 核心内容

### REST基本原则
1. **资源导向** - 将API端点设计为资源而非动作
2. **HTTP方法合理使用** - GET/POST/PUT/DELETE/PATCH
3. **状态码规范** - 使用标准HTTP状态码
4. **版本管理** - API版本控制策略
5. **分页与过滤** - 大数据集处理

### 设计步骤

#### 1. 资源标识与URL设计
```
✓ GET    /api/v1/users              获取用户列表
✓ POST   /api/v1/users              创建新用户
✓ GET    /api/v1/users/{id}         获取特定用户
✓ PUT    /api/v1/users/{id}         完全更新用户
✓ PATCH  /api/v1/users/{id}         部分更新用户
✓ DELETE /api/v1/users/{id}         删除用户

✗ GET    /api/v1/getUserList        不要用动词
✗ POST   /api/v1/createUser         不要用动词
```

#### 2. 请求响应格式
```json
{
  "success": true,
  "data": {},
  "error": null,
  "meta": {
    "timestamp": "2024-01-01T00:00:00Z",
    "version": "1.0"
  }
}
```

#### 3. HTTP状态码规范
- **2xx** - 成功
  - 200: OK（成功）
  - 201: Created（创建成功）
  - 204: No Content（无内容）
- **4xx** - 客户端错误
  - 400: Bad Request（请求错误）
  - 401: Unauthorized（未认证）
  - 403: Forbidden（无权限）
  - 404: Not Found（资源不存在）
  - 422: Unprocessable Entity（验证失败）
- **5xx** - 服务器错误
  - 500: Internal Server Error
  - 503: Service Unavailable

#### 4. 分页实现
```
GET /api/v1/users?page=1&limit=20&sort=-created_at
```

### 常见问题

**Q: 如何处理复杂的业务逻辑？**
A: 不要用URL反映业务逻辑，而是用资源的状态变化表示。
例如：`POST /orders/{id}/cancel` 而不是 `POST /orders/cancelOrder`

**Q: 子资源如何设计？**
A: `GET /api/v1/users/{userId}/posts` - 获取特定用户的文章列表

**Q: 如何版本化API？**
A: 优先在URL中版本化：`/api/v1/` 或 `/api/v2/`
