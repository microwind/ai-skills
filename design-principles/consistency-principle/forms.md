# 一致性原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的代码库一致性如何？

### 快速检查清单

```
□ 相同操作在不同模块中用了不同的方法名（get/fetch/retrieve/find）
□ 错误处理策略不统一（有的返回 null，有的抛异常，有的返回错误码）
□ API 响应格式不统一
□ 日志格式和级别使用不一致
□ 目录结构在各模块间不一致
□ 数据库命名规范不统一（混用 snake_case 和 camelCase）
□ 时间格式不统一（混用 UTC 和本地时区）
□ 配置方式不统一（混用环境变量和配置文件）
□ 代码风格不统一（缩进、括号位置、空行）
```

**诊断标准**:
- 勾选 0-1 项 → **一致性良好**
- 勾选 2-4 项 → **需要建立规范**
- 勾选 5 项以上 → **严重不一致，需要全面整改**

  模块C: □ 返回 null □ 抛异常 □ 返回错误对象 □ 返回错误码

统一策略: ___________________________

### 方法3: API 格式审计

```
当前 API 响应格式：
  端点A: ___________________________
  端点B: ___________________________
  端点C: ___________________________

    "code": ___,

```

### 方法4: 项目结构审计

```

  ├── ___________          ├── ___________

  ├── ___________

  └── ___________
```

## 第3步: 规范制定与执行

### 步骤1: 命名规范

```
约定项:
  类名: □ PascalCase
  方法名: □ camelCase □ snake_case
  常量: □ UPPER_SNAKE_CASE
  数据库表: □ snake_case
  数据库列: □ snake_case
  API路径: □ kebab-case □ snake_case
  文件名: □ kebab-case □ PascalCase

CRUD 操作统一命名:
  创建: ___________
  查询(单个): ___________
  查询(列表): ___________
  更新: ___________
  删除: ___________
```

### 步骤2: 错误处理规范

```
约定:
  输入验证错误: □ 抛 ValidationException
  业务逻辑错误: □ 抛自定义业务异常
  资源未找到: □ 抛 NotFoundException
  系统错误: □ 抛 RuntimeException / 记录日志

  HTTP 错误码映射:
    ValidationException → ___
    NotFoundException → ___
    BusinessException → ___
    其他异常 → ___
```

### 步骤3: 日志规范

```
日志级别使用约定:
  DEBUG: ___________________________
  INFO:  ___________________________
  WARN:  ___________________________
  ERROR: ___________________________

日志格式:
  [时间] [级别] [服务名] [traceId] 消息
```

### 步骤4: 自动化检查

```
工具配置:
  □ Linter（ESLint/Checkstyle/Pylint）配置统一规则
  □ Formatter（Prettier/google-java-format/Black）自动格式化
  □ Pre-commit hook 强制检查
  □ CI 管道自动验证
```

## 常见陷阱预防

### 陷阱1: 新旧代码风格混用
```
✅ 新代码遵循新规范，老代码逐步迁移
❌ 新代码按新规范，老代码不管 → 两种风格并存更混乱
```

### 陷阱2: 一致性 vs 最佳实践
```
✅ 当新的最佳实践明显更优时，全局迁移到新方式
❌ 为了一致性坚持使用过时的做法
```

### 陷阱3: 只有文档没有执行
```
✅ 规范 + Linter + CI + Code Review 多重保障
❌ 写了规范文档但没人遵循
```
