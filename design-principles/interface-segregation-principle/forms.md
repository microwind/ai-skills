# 接口隔离原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的接口是否过于臃肿？

### 快速检查清单

```
□ 实现类中有空方法体
□ 方法实现中抛出 UnsupportedOperationException
□ 实现类只使用了接口中 30% 以下的方法
□ 接口有超过 7 个方法
□ 修改一个方法签名导致大量不相关的类需要重新编译
□ 不同客户端使用同一接口的不同子集
□ 需要通过 null 检查来避免调用某些方法
```

**诊断标准**:
- 勾选 0-1 项 → **接口设计合理**
- 勾选 2-3 项 → **考虑拆分接口**
- 勾选 4 项以上 → **必须拆分，接口严重臃肿**

---

## 第2步: 胖接口识别

### 方法1: 接口使用率分析

```
接口名: ___________________________
方法总数: _____

实现类及其使用的方法:
  实现类A: ___________  使用方法: ___/___ 个 = ____%
  实现类B: ___________  使用方法: ___/___ 个 = ____%
  实现类C: ___________  使用方法: ___/___ 个 = ____%

□ 如果任何实现类使用率 < 50% → 需要拆分
```

### 方法2: 方法分组分析

```
将接口中的方法按功能分组:

组1（___________）:
  □ method1()  □ method2()

组2（___________）:
  □ method3()  □ method4()

组3（___________）:
  □ method5()  □ method6()

□ 如果分组 > 1 → 每组应该是一个独立接口
```

### 方法3: 客户端依赖分析

```
哪些客户端依赖哪些方法:

客户端A（___________）需要: □ method1 □ method2
客户端B（___________）需要: □ method3 □ method4
客户端C（___________）需要: □ method1 □ method3 □ method5

→ 按客户端需求设计接口
```

## 第3步: 接口拆分规划

### 步骤1: 设计新接口

```
原接口: ___________________________（___ 个方法）

拆分为:
  接口1: ___________（___ 个方法）→ 服务于: ___________
  接口2: ___________（___ 个方法）→ 服务于: ___________
  接口3: ___________（___ 个方法）→ 服务于: ___________
```

### 步骤2: 接口组合

```
需要多个能力的场景:
  组合接口: ___________ extends ___________, ___________
```

### 步骤3: 迁移计划

```
第1步: 创建新的小接口（不删除旧接口）
第2步: 让实现类同时实现新旧接口
第3步: 逐步将客户端迁移到新接口
第4步: 删除旧的胖接口
```

---

## 第4步: 测试计划

```
□ 每个新接口的实现类都有独立测试
□ 接口组合正确工作
□ 客户端只依赖需要的接口
□ Mock 更简单（只 mock 需要的接口）
```

---

## 第5步: 代码审查指南

```
□ 每个接口方法数 ≤ 5？
□ 实现类没有空方法或 UnsupportedOperationException？
□ 客户端只依赖它使用的接口？
□ 接口按角色/客户端需求设计？
```

---

## 常见陷阱预防

### 陷阱1: 过度拆分
```
❌ 每个方法一个接口（除非是函数式接口）
✅ 按职责分组，3-5 个方法一个接口
```

### 陷阱2: 忽略接口组合
```
❌ 拆分后需要同时实现 10 个接口
✅ 提供组合接口 (ReadWriteStore extends Readable, Writable)
```

### 陷阱3: 破坏向后兼容
```
❌ 直接删除旧接口
✅ 先标记 @Deprecated，逐步迁移
```

## 重构检查表

```
□ 接口方法数 ≤ 5
□ 无空方法实现
□ 无 UnsupportedOperationException
□ 客户端只依赖需要的接口
□ 有合理的组合接口
□ 所有实现类使用接口 > 80% 的方法
```
