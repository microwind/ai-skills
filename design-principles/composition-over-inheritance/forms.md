# 组合优于继承 - 诊断与规划表

## 第1步: 需求诊断 - 你的继承设计是否合理？

### 快速检查清单

```
□ 继承层次超过 2 层
□ 子类重写了父类方法但行为完全不同
□ 存在多重继承或菱形继承
□ 子类只用了父类 30% 以下的功能
□ 经常需要在父类和子类之间移动代码
□ 添加新功能时需要修改多个子类
□ 子类之间有大量重复代码（父类无法统一）
□ 需要在运行时切换对象的行为
□ 使用了 instanceof 判断具体子类类型
```

**诊断标准**:
- 勾选 0-1 项 → **继承使用合理**
- 勾选 2-4 项 → **应该考虑重构为组合**
- 勾选 5 项以上 → **必须重构，继承设计有严重问题**

### 关系类型判断矩阵

| 关系描述 | is-a（继承） | has-a（组合） | 建议 |
|---------|-------------|-------------|------|
| Cat 是 Animal | ✅ | | 继承合理 |
| Car 有 Engine | | ✅ | 使用组合 |
| Stack 是 ArrayList？ | ❌ | ✅ | 组合更合适 |
| Bird 能 Fly？ | ❌ | ✅ | 组合（能力注入） |
| Button 是 Component | ✅ | | 框架要求，继承合理 |
| AdminUser 是 User | ⚠️ | ✅ | 优先考虑组合+角色 |

## 第2步: 继承问题识别

### 方法1: 继承层次分析

```
当前继承层次：

层级0: ___________________________（基类）
  └─ 层级1: ___________________________
       └─ 层级2: ___________________________
            └─ 层级3: ___________________________

层级深度 = _____ 层

□ 如果 > 2 层 → 考虑用组合拍平层次
```

### 方法2: is-a 关系验证

```
基类: ___________________________
子类: ___________________________

验证问题：
1. 子类是否在所有场景下都是基类的一种？ YES / NO
2. 用子类替换基类后，所有代码都能正确工作？ YES / NO
3. 子类是否需要禁用/空实现父类的某些方法？ YES / NO
4. 这个关系更像"是什么"还是"有什么/能做什么"？ is-a / has-a

□ 如果任何一项为 NO → 改用组合
```

```

1. ___________________________  → 可否通过组合注入？ YES / NO
2. ___________________________  → 可否通过组合注入？ YES / NO
3. ___________________________  → 可否通过组合注入？ YES / NO

需要在运行时切换的行为：
1. ___________________________  → 继承无法实现 → 必须用组合
2. ___________________________  → 继承无法实现 → 必须用组合
```

### 方法4: 脆弱基类检测

```
修改基类时的影响分析：

基类方法: ___________________________
修改内容: ___________________________

受影响的子类：
1. □ ___________________________  → 影响: _________
2. □ ___________________________  → 影响: _________
3. □ ___________________________  → 影响: _________

受影响子类数 = _____ 个

□ 如果修改一个基类方法影响 > 3 个子类 → 耦合过紧，考虑组合
```

## 第5步: 代码审查指南

```
□ 继承合理性
  - is-a 关系是否成立？
  - 继承层次 ≤ 2 层？
  - 子类是否使用了父类的大部分功能？

□ 组合正确性
  - 行为通过接口定义？
  - 通过构造函数/setter 注入？
  - 没有直接 new 具体实现类？

□ 灵活性
  - 行为是否可以运行时替换？
  - 新增行为是否不需要修改现有类？
```

## 常见陷阱预防

### 陷阱1: 把所有继承都改成组合

```
❌ 框架要求的继承也改掉：
  class MyActivity { } // Android 要求继承 Activity

✅ 只重构不合理的继承，保留合理的 is-a 关系
```

### 陷阱2: 组合后委托方法爆炸

```
❌ 为组合对象的每个方法都创建委托：
  getName() → component.getName()
  getAge() → component.getAge()
  ... 20个委托方法

✅ 直接暴露组合对象或使用更高层次的抽象
```

### 陷阱3: 过度接口化

```
❌ 每个方法一个接口
✅ 按行为职责分组，3-5个方法一个接口
```

### 陷阱4: 忽略构造复杂度

```
❌ 组合后构造函数参数过多（> 5个）
✅ 使用 Builder 模式或依赖注入容器
```

### 陷阱5: 没有迁移测试覆盖

```
❌ 重构前没有测试，重构后无法验证正确性
✅ 先补充测试，再重构，测试全通过后删除旧代码
```
