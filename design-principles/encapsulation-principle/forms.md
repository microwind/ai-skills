# 封装原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的封装是否完整？

### 快速检查清单

```
□ 有 public 字段（非常量）
□ getter/setter 暴露了所有内部字段
□ 方法返回了内部可变集合的引用
□ 外部代码直接操作对象内部数据后再调用方法
□ 需要了解对象内部结构才能正确使用它
□ 对象不变量可以被外部代码破坏
□ 构造后对象可能处于无效状态
□ 使用 Map/Dictionary 代替类型化对象传递数据
```

**诊断标准**:
- 勾选 0-1 项 → **封装良好**
- 勾选 2-4 项 → **需要改进封装**
- 勾选 5 项以上 → **严重缺乏封装，立即重构**

## 第4步: 测试计划

```
□ 测试不变量在所有操作后都成立
□ 测试无法通过公开 API 将对象置于无效状态
□ 测试返回的集合不可修改（修改时抛异常）
□ 测试防御性拷贝（修改原对象不影响内部状态）
```

---

## 第5步: 代码审查指南

```
□ 无 public 字段（常量除外）
□ 返回集合时使用不可变视图或副本
□ 外部代码不需要了解内部结构
□ 对象始终处于有效状态
□ 优先使用业务方法而非 getter/setter
□ 可变参数做了防御性拷贝
```

## 常见陷阱预防

### 陷阱1: 返回内部可变集合
```
❌ return this.items;  // 外部可直接修改
✅ return Collections.unmodifiableList(this.items);
```

### 陷阱2: getter/setter 等于公开字段
```
❌ 为每个 private 字段都生成 getter 和 setter
✅ 只暴露必要的业务操作方法
```

### 陷阱3: 通过序列化泄露内部结构
```
❌ 直接序列化领域对象为 JSON（暴露所有字段）
✅ 使用 DTO 控制对外暴露的数据
```

### 陷阱4: 构造不完整的对象
```
❌ new User() 后需要调用多个 setter 才可用
✅ 构造函数强制传入所有必要参数并验证
```

### 陷阱5: 用 Map 代替类型化对象
```
❌ Map<String, Object> userData  // 无编译时检查，无封装
✅ class User { private String name; ... }
```

## 重构检查表

```
□ 所有字段 private（常量除外）
□ 不可变字段使用 final/readonly
□ 集合返回不可变视图
□ 对象构造后即处于有效状态
□ 不变量在所有操作中保持
□ 业务方法替代了 getter+setter 组合操作
□ 无 Map<String, Object> 替代类型化对象
```
