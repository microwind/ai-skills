# 抽象原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的代码是否需要重构？

### 🔍 快速检查清单

```
□ 代码中存在过多的底层实现细节暴露
□ 不同模块之间直接依赖具体实现
□ 修改底层实现需要同时修改上层代码
□ 相似功能在不同地方有不同的实现方式
□ 新功能难以在现有抽象上扩展
□ 代码中存在过深的调用链
□ 接口设计过于具体，难以替换实现
□ 缺乏清晰的分层结构
```

**诊断标准**:
- ✅ 0-1 项 → **很好，抽象层次合理**
- ⚠️ 2-4 项 → **应该考虑重新设计抽象**
- ❌ 5 项以上 → **必须立即重构抽象层**

### 🎯 具体场景评估

| 场景 | 现象 | 建议 | 优先级 |
|------|------|------|--------|
| 数据访问 | SQL 语句散布在业务代码中 | 引入 Repository 抽象 | ⭐⭐⭐⭐⭐ |
| 外部服务 | 直接调用第三方 SDK | 封装 Gateway 抽象 | ⭐⭐⭐⭐ |
| 配置管理 | 硬编码配置在各处 | 提取配置抽象层 | ⭐⭐⭐⭐ |
| 日志系统 | 直接使用具体日志框架 | 统一日志抽象接口 | ⭐⭐⭐ |

## 第3步: 设计抽象层

### 抽象设计策略

```
问题特征 → 抽象策略 → 实现方式

底层细节暴露 → 封装 → 接口 + 实现类
多种实现并存 → 多态 → 抽象类 + 子类
跨层直接依赖 → 分层 → 引入中间抽象层
复杂流程 → 模板方法 → 抽象类 + 钩子方法
算法变化 → 策略 → 策略接口 + 具体策略
```


## 第4步: 迁移规划

### 逐步重构计划

```
阶段1：创建新的抽象层（保持现有代码）
  □ 定义抽象接口
  □ 创建适配实现，包装现有代码
  □ 新功能使用新抽象
  时间：1-2 天
  风险：低

阶段2：逐步迁移调用方
  □ 修改调用方使用新抽象
  □ 保持向后兼容
  □ 逐步替换旧的直接调用
  时间：2-3 天
  风险：中

阶段3：清理旧代码
  □ 删除直接依赖具体实现的代码
  □ 确认所有调用走抽象层
  □ 完整回归测试
  时间：1 天
  风险：中
```

### 验证清单

```
□ 上层代码不直接依赖底层实现
□ 替换底层实现不影响上层逻辑
□ 现有单元测试全部通过
□ 新增测试覆盖抽象层
□ 性能无明显下降
```

## 第5步: 代码审查与评估

### 代码审查清单

```
抽象质量：
□ 每个抽象层只处理一个复杂度级别
□ 抽象接口稳定，不因实现变化而改变
□ 抽象没有泄漏底层实现细节
□ 抽象层次不超过必要的深度

设计质量：
□ 接口方法不超过 7 个
□ 抽象类没有过多的模板方法
□ 无循环依赖
□ 命名准确反映抽象意图

测试质量：
□ 可以通过 Mock 替换底层实现进行测试
□ 抽象层有独立的单元测试
□ 测试覆盖率 ≥ 85%
```

## 常见陷阱预防

### 陷阱1: 过度抽象

```
❌ 过度：每一行代码都包一层抽象
interface StringWrapper { String get(); }
interface IntWrapper { int get(); }
// 没有实际价值的抽象

✅ 合理：仅在变化点和复杂性边界抽象
interface PaymentGateway { Result pay(Amount amount); }
// 有实际价值的抽象
```

**预防**：
```
□ 仅抽象会实际改变的地方
□ 三法则：出现3次才考虑抽象
□ 一个抽象层解决一个问题
```

### 陷阱2: 抽象泄漏

```
❌ 泄漏底层细节：
interface UserRepository {
    User findBySQLQuery(String sql);  // 暴露了SQL细节
}

✅ 隐藏实现细节：
interface UserRepository {
    User findById(Long id);
    List<User> findByName(String name);
}
```

**预防**：
```
□ 接口方法不包含实现相关的参数
□ 返回值不暴露底层数据结构
□ 异常类型是领域层面的
```

### 陷阱3: 错误的抽象层次

```
❌ 太低：抽象过于具体
interface MySQLDatabase { }  // 绑定了具体数据库

✅ 正确：适当的抽象层次
interface DataStore { }  // 通用的数据存储抽象
```
