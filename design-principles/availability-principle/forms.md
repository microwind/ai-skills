# 可用性原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的系统可用性是否达标？

### 快速检查清单

```
□ 系统存在单点故障（单一数据库、单一服务器）
□ 没有健康检查接口
□ 外部服务调用没有超时设置
□ 没有失败重试机制
□ 非核心功能故障会拖垮核心功能
□ 没有监控告警系统
□ 部署需要停机维护
□ 没有灾备方案或恢复演练
□ 日志不足以定位故障原因
□ 没有 SLA 目标定义
```

**诊断标准**:
- 勾选 0-1 项 → **可用性良好**
- 勾选 2-4 项 → **存在风险，需要改进**
- 勾选 5 项以上 → **高危，必须立即改进**

### 可用性等级评估

| 当前状态 | 可用率 | 年停机时间 | 建议 | 优先级 |
|---------|--------|-----------|------|--------|
| 无冗余，手动恢复 | < 99% | > 3.65天 | 立即改进 | ⭐⭐⭐⭐⭐ |
| 有冗余，手动切换 | 99% | 3.65天 | 自动化 | ⭐⭐⭐⭐ |
| 自动故障转移 | 99.9% | 8.76小时 | 优化恢复 | ⭐⭐⭐ |
| 多活架构 | 99.99% | 52.6分钟 | 持续优化 | ⭐⭐ |
| 全球多活 | 99.999% | 5.26分钟 | 保持 | ⭐ |

## 第4步: 测试计划

### 可用性测试

```
1. 故障注入测试
   □ 关闭一个应用实例 → 预期: 请求自动转移到其他实例
   □ 关闭数据库主节点 → 预期: 自动切换到从节点
   □ 断开缓存服务 → 预期: 降级到直接查询数据库
   □ 模拟外部服务超时 → 预期: 断路器打开，返回降级数据
   □ 模拟网络延迟 → 预期: 超时后重试或降级

2. 压力测试
   □ 正常负载 → 响应时间: _____ ms, 错误率: _____ %
   □ 2倍负载 → 响应时间: _____ ms, 错误率: _____ %
   □ 5倍负载 → 响应时间: _____ ms, 错误率: _____ %
   □ 10倍负载 → 预期: 优雅降级而非崩溃

3. 恢复测试
   □ 故障恢复后，系统在 _____ 秒内恢复正常
   □ 积压请求在 _____ 秒内处理完毕
   □ 数据一致性验证通过
```

### 部署验证

```
□ 滚动更新期间零停机
□ 版本回滚在 _____ 分钟内完成
```

## 第5步: 代码审查指南

### 审查检查清单

```
□ 外部调用
  - 所有 HTTP 调用都有超时设置？
  - 有重试机制（带指数退避）？
  - 有断路器保护？
  - 有降级方案？

□ 数据库访问
  - 使用连接池？
  - 有慢查询保护？
  - 读写分离正确？

□ 错误处理
  - 核心功能的错误会导致请求失败？
  - 非核心功能的错误被隔离？
  - 错误日志包含足够上下文？

□ 健康检查
  - /health 端点存在且正确？
  - 检查覆盖所有关键依赖？
  - 不会因检查本身导致问题？

□ 部署
  - 支持滚动更新？
  - 有优雅停机处理（drain connections）？
  - 启动检查（readiness probe）配置正确？
```

## 常见陷阱预防

### 陷阱1: 健康检查过重

```
❌ 健康检查中执行复杂查询：
  /health → SELECT COUNT(*) FROM large_table
  导致检查本身成为性能瓶颈

✅ 轻量级健康检查：
  /health → SELECT 1
  /health/deep → 完整检查（低频调用）
```

**预防**：
```
□ 基础健康检查 < 100ms
□ 深度健康检查有独立端点
□ 健康检查不影响正常业务
```

### 陷阱2: 重试风暴

```
❌ 所有层都无限重试：
  Client 重试 → Gateway 重试 → Service 重试 → DB 重试
  一个请求变成 3^4 = 81 次调用

✅ 分层重试控制：
  只在最靠近故障点的层重试
  加上退避和抖动
  设置全局超时
```

**预防**：
```
□ 只在一个层级重试
□ 有最大重试次数限制
□ 使用指数退避 + 随机抖动
□ 有全局请求超时
```

### 陷阱3: 降级变成常态

```
❌ 降级开关打开后忘记恢复：
  推荐服务降级3个月无人发现

✅ 降级有监控和自动恢复：
  断路器自动 half-open 尝试恢复
  降级持续时间告警
```

**预防**：
```
□ 断路器有自动恢复机制
□ 手动降级开关有过期时间
□ 降级状态有监控告警
```

### 陷阱4: 忽略慢依赖

```
❌ 只处理"挂了"的情况，忽略"慢了"：
  外部服务响应从50ms变成5s，拖慢整个系统

✅ 超时 + 断路器组合：
  超时保证单次请求不会等太久
  断路器在多次超时后快速失败
```

**预防**：
```
□ 所有外部调用都有超时
□ 超时时间 < 用户可感知阈值
□ 有慢调用监控和告警
```

### 陷阱5: 单点故障转移到另一个单点

```
❌ 数据库有主从，但负载均衡器是单点
❌ 服务有多实例，但配置中心是单点

✅ 端到端审查所有环节的冗余性
```

**预防**：
```
□ 从用户请求到数据存储，每个环节都有冗余
□ 基础设施（DNS、LB、配置中心）也有冗余
□ 定期进行端到端故障演练
```

## 改进检查表

完成可用性改进后，使用此检查表验证：

```
架构层面：
□ 无单点故障（所有关键组件有冗余）
□ 故障隔离（一个服务故障不影响其他服务）
□ 自动故障转移（< _____ 秒）
□ 支持零停机部署

应用层面：
□ 所有外部调用有超时（< _____ ms）
□ 关键调用有断路器保护
□ 非核心功能支持降级
□ 有重试机制（带退避策略）

监控层面：
□ 关键指标有实时监控
□ 告警规则覆盖所有关键场景
□ 告警有明确的响应流程
□ 有 On-Call 轮值机制

流程层面：
□ 有故障恢复文档（Runbook）
□ 定期进行故障演练
□ 有事后分析（Post-Mortem）流程
□ SLA 达标率有定期回顾

实际可用率：
□ 上月可用率 = _____%
□ 目标可用率 = _____%
□ 差距 = _____% → 改进计划: _______________
```
