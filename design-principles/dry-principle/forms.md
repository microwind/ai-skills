# DRY原则 - 诊断与规划表

## 第1步: 需求诊断 - 你的代码是否有重复？

### 快速检查清单

```
□ 修改一个业务规则需要改动多处代码
□ 有大段相似的代码出现在不同文件中
□ 相同的验证逻辑在多个地方实现
□ 配置值（魔法数字、阈值）在代码中硬编码且多处出现
□ 多个方法做同样的事，但参数或数据结构略有不同
□ 修了一个 bug，后来发现另一个地方有同样的 bug
□ 数据库字段定义与代码中的模型定义需要手动同步
```

**诊断标准**:
- 勾选 0-1 项 → **DRY 做得好**
- 勾选 2-3 项 → **有重复需要消除**
- 勾选 4 项以上 → **严重重复，需要全面重构**

## 第2步: 重复类型识别

### 类型1: 代码重复（最明显）

```

代码相似度: _____%
共同逻辑: ___________________________
提取为: ___________________________ （方法/类）
```

### 类型2: 逻辑重复（代码不同但做同一件事）

```
实现A: ___________________________
实现B: ___________________________
相同的业务目的: ___________________________
统一实现: ___________________________
```

### 类型3: 知识重复（同一规则多处定义）

```
业务规则: ___________________________
定义位置1: ___________________________
定义位置2: ___________________________
定义位置3: ___________________________
单一权威来源: ___________________________
```

## 第3步: 消除重复规划

### 三次法则评估

```
重复出现次数:
□ 1次 → 直接写，不抽象
□ 2次 → 允许重复，标注 TODO
□ 3次及以上 → 提取共同逻辑

当前重复项:
1. ___________________________  出现 ___ 次 → □保持 □提取
2. ___________________________  出现 ___ 次 → □保持 □提取
3. ___________________________  出现 ___ 次 → □保持 □提取
```

### 消除方法选择

```
□ 提取方法 → 适用于：方法内代码片段重复
□ 提取基类/工具类 → 适用于：类间重复
□ 集中配置 → 适用于：配置值/魔法数字重复
□ 代码生成 → 适用于：数据定义重复（如 OpenAPI → DTO）
□ 模板方法 → 适用于：算法骨架相同，步骤不同
□ 策略模式 → 适用于：业务逻辑相同，策略不同
```

### 执行计划

```

  公共逻辑: ___________________________
  变化部分: ___________________________

第2步：创建共享组件
  类/方法名: ___________________________
  参数: ___________________________
  返回值: ___________________________

第3步：替换所有重复
  位置1: ___________________________ → 调用共享组件
  位置2: ___________________________ → 调用共享组件
  位置3: ___________________________ → 调用共享组件

第4步：验证
  □ 所有原有功能正常
  □ 只需在一处修改即可全局生效
```

---

## 第4步: 测试计划

```
□ 共享组件的单元测试（覆盖所有使用场景）
□ 调用方的集成测试（确保行为不变）
□ 修改共享组件后，所有调用方测试通过
```

---

## 第5步: 代码审查指南

```
□ 新代码是否引入了重复？
□ 是否已有现成的工具方法可以复用？
□ 魔法数字是否抽取为常量？
□ 业务规则是否只有一个定义？
□ 是不是偶然重复？（变化原因不同则允许重复）
```

## 常见陷阱预防

### 陷阱1: 过度 DRY — 错误的抽象
```
❌ 两段恰好相似的代码强行合并，未来它们独立变化时反而更痛苦
✅ 区分偶然相似和本质相同，只合并本质相同的逻辑
```

### 陷阱2: 跨服务共享库过度耦合
```
❌ 微服务间共享大量代码库，导致部署耦合
✅ 接受跨服务的适度重复，优先保证独立部署
```

### 陷阱3: 过早抽象
```
❌ 只出现1次就提取公共方法（猜测未来会复用）
✅ 遵循三次法则，有足够信息再抽象
```

### 陷阱4: DRY 牺牲可读性
```
❌ 为了消除重复，引入过度间接和复杂的泛型
✅ 如果重复代码更清晰易读，适度重复是可接受的
```

## 重构检查表

```
□ 每个业务规则只有一个权威定义
□ 无大段复制粘贴的代码
□ 配置值集中管理
□ 修改一个规则只需改一处
□ 共享组件有完整的测试覆盖
□ 没有为了 DRY 而引入错误的抽象
□ 跨边界的合理重复被保留
```
