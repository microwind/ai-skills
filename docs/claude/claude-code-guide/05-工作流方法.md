# Claude Code 工作流方法：从简单到复杂

工作流是 Claude Code 的核心特性。掌握不同级别的工作流，能处理从小任务到大项目的任何事务。

---

## 工作流的三个级别

### 级别 1：单步工作流（Simple）

最基本的工作流：一个问题，一个答案。

```bash
$ claude "用 Python 写一个 Hello World"

# 输出：完整的 Python 代码
```

**适用**：简单问题、快速查询

**时间**：< 1 分钟

---

### 级别 2：多步工作流（Sequential）

一个大任务分成多个有序步骤。

```bash
$ claude "
步骤 1：设计数据库 → 输出：DDL 脚本
步骤 2：实现后端 API → 输入：数据库设计 → 输出：代码
步骤 3：实现前端页面 → 输入：API 文档 → 输出：React 代码
步骤 4：编写测试 → 输入：前两步的代码 → 输出：测试套件
"
```

**特点**：
- 每一步依赖于前一步的输出
- Claude 记住所有步骤的上下文
- 需要明确的步骤顺序

**适用**：完整项目开发

**时间**：几小时

---

### 级别 3：有状态工作流（Stateful）

跨越多个会话的长期项目，利用恢复会话功能。

```bash
# 第 1 天
$ claude "设计电商系统架构"
# 记下会话 ID: abc123

# 第 2 天
$ claude --resume abc123 "基于架构实现认证模块"

# 第 3 天
$ claude --resume abc123 "现在实现商品模块"

# 第 4 天
$ claude --resume abc123 "写完整的集成测试"
```

**优势**：
- Claude 记住整个项目背景（跨天）
- 建议更连贯、更高质量
- 适合 1-2 周的大项目

**时间**：多天项目

## 工作流设计最佳实践

### 原则 1：明确的分界点

每个步骤应该有清晰的起点和终点。

```bash
❌ 不好
$ claude "
设计、实现、测试这个功能
"

✅ 好
$ claude "
步骤 1：设计数据库结构 → 输出：schema.sql
步骤 2：实现 API 端点 → 输入：schema.sql → 输出：main.go
步骤 3：写单元测试 → 输入：main.go → 输出：*_test.go
"
```

### 原则 2：依赖关系清晰

```bash
❌ 不好
$ claude "
做 A、B、C、D、E
"

✅ 好
$ claude "
步骤 1：A（无依赖）
步骤 2：B（依赖 A）
步骤 3：C（依赖 B）
步骤 4：D, E（都依赖 C）
"
```

### 原则 3：可验证性

每个步骤的输出都应该能验证。

```bash
$ claude "
步骤 1：生成数据库脚本
验证：脚本能否执行无误？

步骤 2：实现代码
验证：代码能编译吗？

步骤 3：运行测试
验证：测试是否全部通过？
"
```

## 实战工作流示例

### 工作流 1：快速功能原型

**场景**：快速验证一个想法

**流程**：
```
需求描述 → 代码实现 → 测试 → 演示
时间：30 分钟
```

**具体提示词**：

```bash
$ claude "
快速原型化一个 Todo 应用前端：

步骤 1：用 React 实现基本功能
- 输入框和 Add 按钮
- Todo 列表显示
- 删除功能

步骤 2：美化样式（Tailwind CSS）
- 响应式设计
- 悬停效果

步骤 3：编写测试
- 添加 Todo 工作吗？
- 删除 Todo 工作吗？

输出：完整的、可运行的应用
"
```

---

### 工作流 2：完整项目开发

**场景**：构建一个新系统

**流程**：
```
需求分析 → 架构设计 → 数据库 → 后端 → 前端 → 测试 → 文档
时间：1-2 天
```

**具体提示词**：

```bash
$ claude "
构建用户管理系统（Java + React）：

步骤 1：设计架构
- 系统组件
- 技术栈选型
- 安全考虑

步骤 2：数据库设计
- User 表（字段、索引）
- Role 表（权限）
- 关系设计

步骤 3：后端实现
- Spring Boot 项目结构
- JPA 实体
- Repository 和 Service
- Controller

步骤 4：前端实现
- React 项目
- 用户列表页面
- 编辑用户页面
- 权限管理页面

步骤 5：完整测试
- 单元测试
- 集成测试
- 页面功能测试

步骤 6：API 文档
- Swagger 文档
- 快速开始指南

每步输出一个独立的、可验证的成果
"
```

---

### 工作流 3：多人协作项目

**场景**：团队开发大型项目

**流程**：
```
架构设计（1 天）
    ↓
[并行]
  后端模块 A（2 天）
  后端模块 B（2 天）
  前端（2 天）
    ↓
集成测试（1 天）
    ↓
部署准备（1 天）
```

**如何用 Claude Code**：

```bash
# 第 1 天：设计架构
$ claude "
设计电商系统架构：
1. 系统组件
2. 数据流
3. API 规范
4. 安全方案
"
# 记下会话 ID: design123

# 第 2-3 天：后端开发者
$ claude --resume design123 "
基于架构设计实现认证模块：
1. 数据库设计
2. JWT 工具类
3. 认证中间件
4. API 端点
5. 单元测试
"

# 第 2-3 天：另一个后端开发者
$ claude --resume design123 "
基于架构设计实现商品模块：
1. 数据库设计
2. 商品 Service
3. API 端点
4. 单元测试
"

# 第 2-3 天：前端开发者
$ claude --resume design123 "
基于 API 规范实现前端：
1. 登录页面
2. 商品列表页面
3. 购物车
4. 支付流程
5. 集成测试
"
```

## 工作流设计清单

设计工作流前，检查以下清单：

- [ ] 目标清晰吗？
- [ ] 每个步骤有明确的输入和输出吗？
- [ ] 依赖关系都列出来了吗？
- [ ] 每个步骤都可验证吗？
- [ ] 步骤顺序正确吗？
- [ ] 时间估算合理吗？
- [ ] 有备选方案吗？

## 常见工作流模式

### 模式 1：渐进式扩展

```
MVP → 功能 A → 功能 B → 性能优化 → 文档
```

适用：快速迭代产品

### 模式 2：分层实现

```
架构 → 数据层 → 业务层 → 表现层 → 集成
```

适用：分层架构项目

### 模式 3：质量驱动

```
需求 → 设计 → 测试 → 实现 → 验证
```

适用：高可靠性项目

### 模式 4：学习驱动

```
理论 → 代码示例 → 动手实现 → 优化 → 总结
```

适用：学习新技术

## 工作流故障排查

### 问题 1：中间步骤失败

```bash
# 如果步骤 B 失败了，怎么办？
❌ 不要从头开始
✅ 恢复到步骤 A，问 Claude 如何修复

$ claude --resume <session-id> "
步骤 B 失败了，错误是...
怎样修复？
"
```

### 问题 2：步骤太长

```bash
❌ 不好
$ claude "实现完整的 API"

✅ 好（拆小）
$ claude "
步骤 1：定义 API 端点
步骤 2：实现登录端点
步骤 3：实现数据获取端点
...
"
```

### 问题 3：输出太多

```bash
❌ 太宽泛
$ claude "生成整个项目"

✅ 具体
$ claude "
生成项目的：
- 目录结构
- go.mod 依赖
- main.go 框架
"
```
