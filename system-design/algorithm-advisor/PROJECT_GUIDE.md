## 💡 关键决策点

| 决策 | 选项 | 建议 | 理由 |
|------|------|------|------|
| **算法选择** | 仅CF / 仅Content / 混合 | 混合 | 准确率+多样性最优 |
| **缓存策略** | 单层 / 多层 | 三层 | 命中率>80% |
| **模型** | 协同过滤 / DNN | CF+CTR | 性能和准确率平衡 |
| **特征更新** | 离线 / 实时 | 双层 | 实时性 + 效率平衡 |
| **扩展方式** | 单机 / 分布式 | 分布式 | 支持10倍增长 |
| **部署方式** | Docker / K8s | K8s | 自动扩缩容 |

## 🔧 常见问题

### Q1: 系统如何支持 10 倍增长?
```
A: 四个方向同时扩展
  1. 应用层: 5-7 台 → 20-30 台 (水平扩展)
  2. 缓存层: 100GB → 400GB (更多 Redis)
  3. 离线层: Spark 集群扩大 + GPU (并行度提升)
  4. 数据库: 分片处理 (按 user_id)

详见: scalability-and-optimization.md
```

### Q2: 冷启动怎么处理?
```
A: 三层降级方案
  1. 新用户: 内容推荐 + 热度排序
  2. 新商品: 内容相似 + 热度
  3. 无历史: 协同过滤用其他用户

详见: hybrid-recommendation-system.md
```

### Q3: 如何快速迭代优化?
```
A: A/B 测试框架
  1. 并行运行多个实验
  2. 自动流量分配
  3. 自动统计检验
  4. 自动化报告

详见: monitoring-and-testing.md
```

### Q4: 故障如何应对?
```
A: 多层容错方案
  1. Redis 故障: 在线计算降级
  2. MySQL 故障: 缓存/本地队列
  3. 模型故障: 使用融合得分
  4. 网络故障: 本地缓存返回

详见: deployment-guide.md#故障处理
```
